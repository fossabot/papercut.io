# Papercut v1.1.0 — Release Notes

- **Target release:** 1.1.0
- **Baseline:** [`v1.0.0`](https://github.com/muhannadnouri/papercut.io/releases/tag/v1.0.0) (commit `9062e6e`, tagged 2026-06-04)
- **Range covered:** 2026-06-04 → 2026-06-22

- **Release branch:** the 1.1.0 version bump, tagging, and release artifacts will be produced on a new `branch-release-v1.1.0` branch cut from `main`; these notes are the pre-release changelog for that work.

This release turns Papercut's English-only Kokoro desktop reader into a multilingual, mobile-first audiobook app. The five merged PRs add native mobile background playback, a catalog-backed multilingual TTS architecture (Arabic Piper + Libtashkeel diacritization), device-aware thread tuning, a touch-friendly audio UI, large-book performance work, and a long list of disk- and duration-correctness fixes. Existing v1.0.0 audiobooks remain playable without re-save.

## Highlights

- **Native mobile audiobook playback.** Saved audiobooks now play through Android Media3/ExoPlayer via the pinned `tauri-plugin-native-audio` 1.0.5, with background and lock-screen playback. Mobile prepares one reusable seekable `playback.wav` instead of streaming per-chunk blobs, which eliminates per-chunk Blob URL lifecycle cost and keeps long-book audio alive when the WebView is backgrounded (`src-tauri/src/native_tts/engine/playback.rs:36`, `src/tts/playback/nativeMobileAudio.ts:24`).
- **Multilingual TTS.** A catalog-backed model architecture adds Piper Kareem Medium (`ar-JO`) alongside Kokoro. Arabic-dominant documents auto-suggest Piper; users can override the model selector (`src-tauri/tts/model-manifest.json:16`, `src-tauri/src/native_tts/engine/models.rs:38`).
- **Arabic diacritization.** Optional Libtashkeel 1.5.0 preprocessing restores short vowels on the synthesis copy only — source chunks, highlighting, search, and bundle metadata are never rewritten (`src-tauri/src/native_tts/engine/preprocess.rs`).
- **Device-aware TTS threads.** Native ONNX Runtime thread counts are now tuned per platform (Android conservative, desktop follows CPU) and exposed in the Audio Setup panel with a high-thread warning. Right-sizing threads avoids the thermal-throttling regime where more threads actually slows synthesis end-to-end (`src-tauri/src/native_tts/platform.rs:11`, `src/tts/components/AudioSetupPanel.tsx:152`).
- **Large-book performance.** Rapid chunk navigation is coalesced and serialized, highlighting is bounded to a current-chunk LRU range cache, and the chunk menu is virtualized — keeping CPU and memory flat on long books instead of scaling with chunk count (`src/tts/hooks/useTtsPlayer.ts:69`, `src/tts/hooks/useTtsHighlight.ts:219`).
- **Touch-friendly audio UI.** The fixed speed dropdown is replaced by a 0.5x–2.0x slider with coarse-pointer hit targets, and the Audio Setup panel layout and labels are polished for mobile (`src/tts/components/AudioSetupPanel.tsx:127`).
- **Linux AppImage audiobook playback fixed.** The AppImage now bundles the GStreamer media framework WebKitGTK needs for WAV playback, with a CI verification step before publishing (`scripts/verify-appimage-media.js:21`, `src-tauri/tauri.linux.conf.json`).

## Added

<summary><strong>6 new capabilities</strong> — native mobile playback, catalog-backed multilingual TTS, Arabic diacritization, device-aware threads, touch-friendly speed control, silent placeholder for empty chunks</summary>

### Native mobile playback and reliable cleanup
- Prepare a reusable single-track `playback.wav` for saved audiobooks so Android can seek and play in the background and on the lock screen (`src-tauri/src/native_tts/engine/playback.rs:36`, `src-tauri/src/native_tts/engine/paths.rs:96`).
- Background and lock-screen playback through the official, exactly pinned `tauri-plugin-native-audio` 1.0.5 packages. Papercut serializes bridge commands and owns its foreground polling cadence rather than forking the plugin (`src/tts/playback/nativeMobileAudio.ts:24`).
- Coalesce rapid chunk navigation and preserve highlighting across skips; serialize native source and playback lifecycle operations so Stop and Back no longer fail on Tauri listener handle mismatch.
- Allow native audio initialization retries so a transient plugin/bridge startup failure does not wedge playback for the session.
- **Performance implication:** desktop reads a bounded window of saved chunk WAVs through Blob URLs, revoking out-of-window URLs; mobile instead maps one native `playback.wav` timeline back to chunk-local state via binary-searched boundaries. This removes per-chunk Blob creation/revocation, avoids holding many large Blob references in WebView memory, and lets audio outlive the backgrounded WebView (the native Media3 session owns playback, not the WebView). The trade-off is a one-time stitch cost at first Play unless an imported track already exists; afterward seeks are O(1) native file seeks rather than per-chunk assembly.

### Catalog-backed multilingual TTS
- A generic Rust model catalog is now the authoritative source of model metadata, required-file validation, family loaders, voices, and preprocessing capabilities. A matching TypeScript fallback keeps browser/stub UI deterministic (`src-tauri/src/native_tts/engine/models.rs:38`, `src/tts/models.ts:12`).
- **Piper Kareem Medium (`ar-JO`)** added through sherpa VITS — one voice, 67,177,830-byte archive, downloaded on demand from the pinned `k2-fsa/sherpa-onnx` TTS-models release and SHA-256 verified (`src-tauri/tts/model-manifest.json:16`).
- Arabic sentence and clause punctuation is recognized during chunking, and every synthesis request has a hard character bound to reduce native crashes on long unpunctuated text.
- The model/voice/preprocessing selector is exposed in React; Arabic-dominant documents auto-suggest Piper, overridable by the user (`src/tts/components/AudioSetupPanel.tsx`, `src/tts/models.ts:81`).

### Arabic diacritization (Libtashkeel 1.5.0)
- Shared native builds compile `libtashkeel_base = 1.5.0` as an optional preprocessing backend that runs through the same packaged ONNX Runtime as sherpa-onnx (`src-tauri/src/native_tts/engine/preprocess.rs`).
- Two options are advertised per model: `none` (pass source text unchanged; the only Kokoro option) and `libtashkeel-1.5.0` (Piper default on desktop shared builds and Android).
- Diacritics are applied only to the synthesis copy. Canonical `chunk.text`, chunk IDs, source hashes, source spans, the HTML, the search index, and DOM range matching are untouched, so highlighting stays aligned to the original document.
- **Performance implication:** Libtashkeel runs once per missing chunk on the synthesis string only, on the native save thread; it reuses the same ONNX Runtime sherpa-onnx already loads, so it adds no second inference runtime to the bundle/APK and does not touch the WebView. Because the source text and spans are never rewritten, highlighting's chunk-based range lookups keep their O(1)-amortized cost.

### Device-aware native thread controls
- ONNX Runtime thread counts are tuned per platform: Android is conservative (capped at 4) to avoid thermal throttling, desktop follows `available_parallelism` (`src-tauri/src/native_tts/platform.rs:11`).
- Thread count is exposed in the Audio Setup panel with a detected max, applied-count readback, and a high-thread warning about memory, heat, and battery drain (`src/tts/components/AudioSetupPanel.tsx:152`).
- `resolve_thread_count` clamps a caller-requested value into the detected CPU range, falling back to the platform default when unset or non-positive (`src-tauri/src/native_tts/platform.rs:28`).
- **Performance implication:** more threads is not always faster for ONNX inference. On Android, high thread counts trigger thermal throttling, battery drain, and memory pressure that can make synthesis slower end-to-end than a conservative count. Capping Android at 4 and desktop at `available_parallelism` sustains throughput instead of bursting and throttling. The engine rebuilds only when the desired thread count differs from the loaded one, so changing threads is a controlled one-time reload rather than a per-chunk cost (`src-tauri/src/native_tts/engine/synth.rs:48`).

### Touch-friendly speed control
- Replace the fixed 0.85x/1x/1.1x/1.2x speed dropdown with a range slider spanning 0.5x–2.0x in 0.05 steps, flanked by − / + step buttons and a live value readout (`src/tts/components/AudioSetupPanel.tsx:127`).
- `snapSpeed` snaps each change to the step, clamps to range, and rounds to two decimals so the JS cache id (`toFixed(2)`) and Rust id (`{speed:.2}`) stay byte-stable; all previous discrete values land exactly on the new grid (`src/tts/components/AudioSetupPanel.tsx:13`).
- Under `@media (pointer: coarse)` the thumb grows to ~26px, the step buttons to 44px, and the row to a 44px hit height; `touch-action: none` stops the scrollable setup panel from hijacking horizontal drags.
- **Performance implication:** because `snapSpeed` keeps cache ids byte-stable across the JS/Rust boundary, changing speed never causes a spurious cache miss or accidental full-book regeneration; existing saved records reload and reuse their WAVs unchanged.

### Silent placeholder for empty chunks
- A chunk whose text is empty after sanitization (e.g. a paragraph of only emoji or other non-BMP symbols the sanitizer drops) now writes a 0.25s silent WAV instead of aborting the entire audiobook save. The placeholder is written through sherpa-onnx's own WAV writer so its encoding is byte-identical to generated chunks and single-track export concatenation stays valid (`src-tauri/src/native_tts/engine/synth.rs:160`, `src-tauri/src/native_tts/engine/synth.rs:150`).
- **Performance/reliability implication:** previously a single empty chunk aborted the whole save and resume re-hit the same chunk and failed identically, wedging the book so no further progress was possible. The placeholder lets the save loop continue, so one bad chunk can no longer block an entire long book. Because the placeholder's encoding is byte-identical to real chunks, single-track stitching needs no transcoding or special-casing at export or mobile-playback-prepare time.

## Changed

<details>
<summary><strong>6 changes</strong> — CSS Custom Highlight API refactor, year-expansion gating, canonical save duration, number-words refactor, HTML narration extraction, Audio Setup panel polish</summary>

### Highlighting refactored to the CSS Custom Highlight API
- TTS highlighting was rewritten on top of the CSS Custom Highlight API, replacing the prior DOM-wrapping approach. The net change is roughly −200 lines, and the new path registers one named `Highlight` and rebuilds only the current chunk's `Range` (`src/tts/hooks/useTtsHighlight.ts:178`, `src/tts/hooks/useTtsHighlight.ts:195`, `src/tts/hooks/useTtsHighlight.ts:319`).
- Highlight cleanup is now guarded during iframe teardown so a navigation away from the document cannot clear a stale registry on the wrong `Document` (`src/tts/hooks/useTtsHighlight.ts:319`).
- **Performance implication:** the prior DOM-wrapping approach mutated the document (wrapping text nodes in highlight spans), which forces reflow and layout invalidation on large documents and interferes with iframe sizing and in-document Find. The CSS Highlight API registers `Range` objects in a registry without touching the DOM tree, so there is no reflow, no layout thrash, and Find/selection stay unaffected. Combined with current-chunk-only `Range` creation and a bounded LRU range cache, the per-navigation cost is O(1) range builds amortized and memory is bounded by `MAX_CACHED_RANGES` rather than the chunk count — which is what makes highlighting viable on thousand-chunk books.

### English year expansion gated and tightened
- Year expansion moved out of the shared `sanitize_tts_text` (which every model calls) into `generate_audio`, applied only when the model opts in via `ModelDefinition::expands_english_years` (Kokoro family + English language). The shared sanitizer is now pure, language-neutral Unicode cleanup, so non-English models (e.g. Arabic Piper) no longer have English number words spliced into their audio (`src-tauri/src/native_tts/engine/synth.rs:203`, `src-tauri/src/native_tts/engine/models.rs:87`).
- The heuristic was tightened: a four-digit run reads as a year only when both sides look like prose (whitespace or opening/closing punctuation). Decimals, currency amounts, percentages, digit-grouped numbers, hyphen ranges, and surrounding identifiers are no longer expanded as years (`src-tauri/src/native_tts/engine/synth.rs:275`, `src-tauri/src/native_tts/engine/synth.rs:315`).
- Synthesis-text only: source chunks, highlighting, search, bundle metadata, and cache identity are untouched.
- **Performance implication:** moving year expansion out of the shared sanitizer means non-English models skip the year-scanning regex entirely on every chunk; the shared sanitizer is now pure Unicode cleanup, cheaper on every synthesis path. The opt-in check is a single family/language match, not a text scan.

### Canonical audiobook save duration
- The save response and final saved event now report `audio_duration_sec` from the canonical f64 totals the manifest writer measures from WAV headers via `build_playback_index`, instead of a per-chunk f32 running accumulation that drifted on long books. The response field widened from f32 to f64 (transparent over IPC). Live mid-save progress stays f32 since it is display-only (`src-tauri/src/native_tts/engine/save.rs:425`).
- **Performance/correctness implication:** the f32 running sum drifted from the canonical f64 total over long books, so saved-record duration and byte totals could disagree with the manifest and the on-disk WAVs. Reading the measured total back from `write_manifest` removes the drift without adding a second pass — the totals are already computed when the playback index is built.

### Number-words composition refactor
- `below_hundred_words`, which previously enumerated all 100 values 0–99 as one match arm each, is now composed from two small lookup tables (`ONES` 0–19, `TENS`), cutting ~105 lines to ~20. Behavior is identical and remains covered by the `year_expansion_*` tests (`src-tauri/src/native_tts/engine/synth.rs:385`).

### HTML narration extraction
- The HTML narration adapter now preserves prose placed directly inside readable wrappers such as legacy table cells, even when those wrappers also contain nested headings or paragraphs. A generic Arabic HTML fixture covers this pattern (its first paragraph is a direct `td` text node, the next is inside `p`); the former extractor omitted the direct text before Piper received any chunk (`src/tts/alignment/htmlStructure.ts:63`).
- Bracketed inline footnote reference links such as `[1]` and `[2*]` are skipped during narration extraction and DOM highlight indexing, while the actual footnote paragraphs remain readable later in the document.

### Audio Setup panel UI polish
- Reorder controls so the universal Model, Voice, Speed, and Threads selects come first and the conditional Text-processing select renders last, so the layout no longer jumps when the selected model changes. Top-align the settings grid so a field with wrapping helper text no longer offsets the row.
- Shorten the Arabic preprocessor label "Automatic Arabic diacritization" → "Auto diacritization" (the full explanation already lives in the description line beneath it).
- Show a check icon instead of a download icon once the voice model is installed; rename the Kokoro model "Kokoro English v1.0" → "Kokoro v1.0" (the option label already prefixes the language); normalize `<select>` chrome with `appearance:none`, a custom caret, and an explicit white background so WebKitGTK host theming no longer greys some selects; track the setup gear tooltip to its open/closed state; raise model source attribution contrast to meet WCAG AA for small text (`src/tts/components/AudioSetupPanel.tsx`, `src/tts/components/AudioControls.tsx`).

### Navigation and highlighting stability
- Stabilize rapid chunk navigation and highlighting so fast skips no longer desynchronize the highlight from the playing chunk or leave stale ranges behind (`src/tts/hooks/useTtsPlayer.ts:115`, `src/tts/hooks/useTtsHighlight.ts:35`).
- Guard highlight cleanup during iframe teardown so an iframe navigation cannot clear a highlight registry on the wrong `Document` (`src/tts/hooks/useTtsHighlight.ts:319`).

### Native audio initialization
- Allow native audio initialization retries so a transient `tauri-plugin-native-audio` bridge startup failure does not wedge playback for the whole session.

### Linux AppImage audiobook playback
- Fix Linux AppImage audiobook playback by bundling the GStreamer media framework WebKitGTK uses for WAV playback. The Linux CI and release builds now install the required GStreamer plugins and `patchelf`, and `verify-appimage-media` checks the AppImage for the WAV parser, plugin scanner, core elements, and an audio sink (ALSA, PipeWire, or PulseAudio) before publishing (`scripts/verify-appimage-media.js:21`, `src-tauri/tauri.linux.conf.json`).
- Build Linux packages on Ubuntu for broader compatibility and document the local GStreamer prerequisites.

### Imported audiobook playback metadata
- Preserve imported audiobook playback metadata. Imported bundles now replay from their stored bundle chunk metadata instead of re-chunking the restored source HTML, so older completed WAV audiobooks remain playable even when newer extraction or chunk-boundary code would produce a different source signature. Highlight spans for imported bundles are rebuilt lazily from restored HTML and grafted only when the rebuilt chunk ids/text exactly match the bundle chunks; playback is not blocked while that work runs.

### HTML narration chunk/highlight alignment
- Align HTML narration chunks with highlights so the highlighted source range tracks the chunk actually being synthesized, including the newly retained wrapper-level prose.

### Orphan and stale chunk WAV reclamation
- Sweep orphan chunk WAVs on audiobook re-save. Chunk WAV filenames embed a content hash, but the cache directory is keyed only by document URL + playback options, so editing a document and re-saving wrote new chunk files into the same directory while stale ones lingered forever. `prune_orphan_chunk_files` now runs once every current chunk is generated, before the manifest is finalized. Resume is unaffected because the sweep runs only after successful generation and keeps every expected chunk (`src-tauri/src/native_tts/engine/save.rs:170`, `src-tauri/src/native_tts/engine/save.rs:633`).
- Reclaim stale chunk WAVs *before* re-save as well. Editing a document's source and re-saving reuses the same audiobook id (its hash omits chunk content), so the on-disk chunk dir is shared across generations. Orphans from the previous text were only swept after the new save completed; an abandoned re-save then left the stale set indefinitely. A second up-front sweep now runs right after the pending manifest is written and before generation (`src-tauri/src/native_tts/engine/save.rs:170`).
- Keep in-flight chunk temps during the orphan sweep. `synthesize_to_file` writes each chunk to `<name>.<nonce>.tmp` and atomically renames it into place, so a concurrent save of the same audiobook id can have a live temp mid-write. The start-of-job sweep runs before the engine lock is taken, so it raced the other job's generation and could delete its temp, failing the commit rename. `.tmp` entries are now skipped in the sweep (`src-tauri/src/native_tts/engine/save.rs:633`).
- Reclaim abandoned chunk temp files on save. A process that dies mid-write leaves a `.tmp` behind; the unique nonce means a later attempt never overwrites it and the orphan sweep skipped `.tmp` on purpose, so the remnant lingered forever. `prune_stale_temp_files` runs in the final sweep with a job-start walltime cutoff: temps last modified before this job began are abandoned and removed, while a concurrent save's in-flight temp — necessarily written after this job started — is kept. Temps with unreadable mtime are left alone rather than risk deleting a live write (`src-tauri/src/native_tts/engine/save.rs:672`).
- **Performance/disk implication:** without these sweeps, repeated re-saves of edited documents leaked disk indefinitely (stale chunk WAVs with outdated content-hash names lingered forever in the shared cache dir) and inflated the byte totals reported on delete. The up-front + final sweeps bound on-disk bytes to the current chunk set, keep delete byte totals accurate, and the `.tmp`-aware, job-start-cutoff design makes the sweeps safe to run concurrently with another save of the same id. Abandoned-temp reclamation prevents the nonce-named `.tmp` files (which a later attempt never overwrites) from accumulating forever.

### Empty-chunk save aborts
- A chunk whose text was empty after sanitization previously made `generate_audio` return an error, which propagated through the save loop and aborted the entire audiobook save. Resume re-hit the same chunk and failed identically, wedging the book permanently. `generate_audio` now returns `Ok(None)` for empty-after-sanitization text and `synthesize_to_file` writes a 0.25s silent placeholder instead of failing. Genuine synthesis failures still error (`src-tauri/src/native_tts/engine/synth.rs:160`).

## Performance

This section consolidates the throughput, memory, and responsiveness work. The unifying goal is that CPU, memory, and bridge traffic stay flat as books get longer, rather than scaling with chunk count.

### Bounded navigation and highlighting work
- **Latest-target-wins mobile navigation.** Intermediate queued targets are replaced before commit, so hammering next/prev bounds rapid skips to at most one in-flight native bridge command instead of queueing unbounded IPC or audio source swaps (`src/tts/hooks/useTtsPlayer.ts:511`).
- **Coalesced tap handling.** Taps within one animation frame are coalesced before starting the serialized seek worker, so a burst of skips becomes a single navigation intent (`src/tts/hooks/useTtsPlayer.ts:630`).
- **Serialized native audio commands.** Source and playback lifecycle operations (setSource/play/seek/stop) are serialized so overlapping bridge calls no longer cause listener handle mismatch and Stop/Back failures.
- **Bounded 250 ms mobile progress polling.** Foreground polling is bounded to 250 ms instead of a tight loop, capping bridge traffic and foreground CPU during mobile playback; foreground synchronization after screen lock recovers state without re-polling stale data (`src/tts/hooks/useTtsPlayer.ts:69`).
- **Virtualized chunk menu and memoized document iframe.** Navigation through long books no longer re-renders the world: the chunk menu renders only visible rows (a 1000-chunk book does not mount 1000 DOM nodes), and the document iframe is memoized so unrelated state changes do not re-parse/re-render the document.
- **Source-span highlighting with current-chunk-only DOM range creation.** Only the playing chunk's `Range` is built on demand, backed by a bounded LRU range cache with memoized failed mappings to avoid repeated DOM scans and duplicate diagnostics. Recent ranges are reused, oldest evicted, so memory is bounded by `MAX_CACHED_RANGES` rather than chunk count (`src/tts/hooks/useTtsHighlight.ts:219`).
- **Stale-operation fencing.** In-flight operations are rejected by job/navigation ids and never commit obsolete audio, so there is no wasted work rendering stale state after a newer navigation overrides it (`src/tts/hooks/useTtsPlayer.ts:482`).

### Large audiobook storage and playback
- Mobile prepares one reusable local `playback.wav` for background/lock-screen playback rather than streaming per-chunk blobs, and the import path can restore `playback.wav` without a device-local sidecar (`src-tauri/src/native_tts/engine/playback.rs:36`, `src-tauri/src/native_tts/engine/playback.rs:110`).
- **Performance implication:** the single-track stitch is a one-time concatenation at first Play; thereafter seeks are native O(1) file seeks, not per-chunk Blob assembly. This avoids the WebView holding N Blob URLs and their memory, and lets the native Media3 session own playback independently of the WebView lifetime. Desktop keeps its bounded-window Blob URL strategy (with out-of-window URL revocation) since it does not need background playback; mobile's single-track strategy is what makes long-book background playback viable within Android's memory budget.

## Compatibility and Upgrade Notes

This is a minor version bump (1.0.0 → 1.1.0). No breaking changes to saved user data are intended; the changes below are the compatibility surface maintainers and upgraders should know about.

- **Cache version unchanged.** `CACHE_VERSION` remains `native-save-v4-segmented`; no global invalidation is triggered (`src-tauri/src/native_tts/engine/config.rs:15`, `src/tts/types.ts:3`).
- **Kokoro identity unchanged.** Kokoro keeps its exact model ID (`sherpa-onnx/kokoro-multi-lang-v1_0`), voice IDs, and audiobook cache key, so existing Kokoro audiobooks reload unchanged.
- **Legacy records default safely.** Legacy preferences, download records, saved records, imported uploads, native manifests, and bundles without model metadata default to Kokoro; those without preprocessing metadata default to `none`. Existing completed WAV audiobooks remain playable.
- **Diacritized Piper gets a distinct audiobook ID.** Audiobook identity includes the versioned preprocessing ID only when it is not `none`, so diacritized Piper saves get a separate cache directory, saved record, queue entry, manifest, and export-bundle field and cannot silently reuse older undiacritized WAV chunks. Existing Kokoro and undiacritized audiobook IDs remain byte-for-byte unchanged (`src-tauri/src/native_tts/engine/paths.rs:225`).
- **Bundle format unchanged.** `BUNDLE_MAGIC` is still `PAPERCUT_AUDIOBOOK_BUNDLE_V2\n` and `AUDIOBOOK_MANIFEST_VERSION` is still `2`. Imported bundles replay from their stored chunk metadata, so older Kokoro bundles remain playable even when newer chunk-boundary code would produce a different source signature (`src-tauri/src/native_tts/engine/config.rs:18`).
- **Year expansion is synthesis-only.** Moving year expansion out of the shared sanitizer and gating it on `expands_english_years` does not alter source chunks, highlighting, search, bundle metadata, or cache identity. English Kokoro years still expand; everything else keeps its original digits.
- **Wrapper-text documents must be regenerated.** Documents affected by the wrapper-level prose omission (e.g. direct `td` text nodes) must be regenerated to include the newly retained prose. The corrected source signature and chunk sequence intentionally do not match the incomplete generation, so the old audio is not silently reused. Most books produce identical chunks and are unaffected.
- **No `CACHE_VERSION` bump for the re-save sweep changes.** The up-front + final orphan sweep deliberately does not bump `CACHE_VERSION`. The text-owner segmentation change alters chunk boundaries only for documents with wrapper-level prose; those self-heal via source-signature mismatch and re-save. A version bump would force every unchanged book to regenerate, which the docs intentionally avoid.
- **Speed slider is cache-compatible.** All previous discrete speed values (0.85x/1x/1.1x/1.2x) land exactly on the new 0.05-step grid, and `snapSpeed` keeps the JS/Rust cache ids byte-stable, so existing saved records reload unchanged.
- **Android minimum API level.** Native playback uses Android Media3/ExoPlayer through `tauri-plugin-native-audio` 1.0.5; API 26 remains the minimum supported Android version.

<a id="version-bump-workflow"></a>
### Version-bump workflow

The 1.1.0 release work happens on a new `branch-release-v1.1.0` branch cut from `main` (not directly on `main`). On that branch, follow the README version-bump checklist and keep these in sync: `package.json`, `package-lock.json`, `src-tauri/tauri.conf.json`, `src-tauri/Cargo.toml`, `src-tauri/Cargo.lock`. Then run `cargo check --manifest-path src-tauri/Cargo.toml --features native-tts-shared` and `npm run build`, commit the version files, and tag `v1.1.0` from the release branch.
